What Composition over Inheritance Means

I recently stumbled upon a promising Ruby web framework called Hanami, whose description pitch immediately grabbed me. “We havent made the mistakes of that other framework. We’ve designed from scratch with the principles of the Clean Architecture and Domain Driven Design in mind!”

First off, we use composition over inheritance! So instead of

class FooController < ActiveController
  # ... 
end

you write

class FooController
  include Hanami::Controller
  # ...
end

A brief aside

At my day job, there is a module that aims to aid in helping to test a particularly complicated set of entities. Using that interface typically looks something like this (names redacted).

setup do
  setup_foo_helper(name: 'Something')
end

test 'field has the right value' do
  assert_field 'some_field', 10
end

Now, do you have any idea what this code does? What behaviour this is testing? Where these methods are even coming from? Well no, and neither do I. This is the problem that is addressed by prefering composition to inheritance. Disambiguation, and easy of understanding.
